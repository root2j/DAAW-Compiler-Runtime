## Workflow Title: Automated Welcome Email and CRM Integration
## Overview: This workflow automates the post-submission process for a Google Form. Upon a new submission, it will extract user data, send a personalized welcome email via Gmail, and integrate the user's information into a Customer Relationship Management (CRM) system.       
## Trigger: New Google Form submission.

## Subtask Breakdown

### Phase 1: Trigger and Initial Data Extraction
**Type:** Sequential
**Depends on:** Nothing
**Subtasks:**
1.  **Listen for Google Form Submission Event:** The workflow system will be configured to actively listen for or receive webhook notifications for new form submissions from a specified Google Form.
2.  **Extract and Standardize Form Submission Data:** Parse the incoming event payload to extract all relevant submitted fields. This includes, but is not limited to:
    *   User's Full Name (or separate First Name, Last Name)
    *   User's Email Address
    *   Any other data points required for the welcome email personalization or CRM entry.
**Output of this phase:** A structured data object containing all extracted form fields, with key fields like user's name and email address clearly identified.

### Phase 2: Send Welcome Email
**Type:** Sequential
**Depends on:** Phase 1
**Subtasks:**
1.  **Prepare Welcome Email Content:**
    *   Retrieve the pre-defined welcome email template.
    *   Dynamically populate placeholders within the template using the user's name and any other relevant data extracted in Phase 1.     
2.  **Initiate Email Send via Gmail:**
    *   Make an API call to the Gmail service to send an email.
    *   Specify the recipient's email address using the email extracted in Phase 1.
    *   Specify the sender's email address (configured Gmail account).
    *   Attach the prepared email subject and body content.
3.  **Monitor Email Send Status:** Await the response from the Gmail service to confirm the email send operation's success or failure.    
    *   **On Success:** Record the successful email dispatch.
    *   **On Failure:** Capture the error details (e.g., invalid recipient, authentication issue) and flag for error handling.
**Output of this phase:** Status of the email send operation (success/failure) and any associated error messages or success confirmations.

### Phase 3: CRM Integration
**Type:** Sequential
**Depends on:** Phase 1
**Subtasks:**
1.  **Map Form Data to CRM Fields:** Transform the extracted form data from Phase 1 into the specific field names and data types required by the target CRM system's API (e.g., "Full Name" from form to "firstName" and "lastName" in CRM, "Email Address" to "email").
2.  **Authenticate with CRM System:** Ensure that the API call to the CRM system includes valid authentication credentials (e.g., API key, OAuth token).
3.  **Create/Update Contact Record in CRM:**
    *   Make an API call to the CRM system to create a new contact record using the mapped user data.
    *   Pass all necessary user information (name, email, etc.) to the CRM API.
    *   **On Success:** Record the successful CRM entry, including any ID returned by the CRM.
    *   **On Failure:** Capture the error details (e.g., API error, data validation failure) and flag for error handling.
**Output of this phase:** Status of the CRM operation (success/failure), the CRM contact ID (if created), and any associated error messages or success confirmations.

### Phase 4: Error Handling and Notifications
**Type:** Grouped (triggered by failures in Phase 2 or Phase 3)
**Depends on:** Phase 2 (failure path), Phase 3 (failure path)
**Subtasks:**
1.  **Identify Error Source and Context:** Determine precisely which operation failed (email send or CRM integration) and retrieve all available error messages and the original input data that caused the failure (e.g., user's email, full form submission).
2.  **Log Detailed Error Information:** Record the specific error message, timestamp, affected user's data, and the originating phase of the error to a designated internal logging system or database.
3.  **Send Internal Failure Notification:**
    *   Compose a clear and concise notification message detailing the error (e.g., "Welcome email failed for [user_email]", "CRM contact creation failed for [user_name]").
    *   Send this notification to a pre-defined internal channel (e.g., administrative email address, Slack channel, Microsoft Teams).    
**Output of this phase:** Error logged, internal notification sent.

## Conditional Logic & Edge Cases
*   **Independent Failure Handling:** The workflow must be designed such that a failure in sending the welcome email (Phase 2) does not prevent the attempt to integrate with the CRM (Phase 3), and vice-versa. Both operations should proceed independently, with their respective failures routed to Phase 4.
*   **Specific Error Routing:** Phase 4 must be able to differentiate between an email sending error and a CRM integration error to provide context-specific logging and notifications.
*   **No Complex Business Logic:** As per the brief, there is no conditional logic beyond handling explicit operational failures. For example, there is no logic to check for existing CRM contacts before creation, or to send different emails based on form content.

## Data Flow Summary
1.  **Google Form Submission:** Initiates the workflow with raw form field data.
2.  **Phase 1 (Trigger & Extraction):** Transforms raw form data into a standardized, structured JSON object containing user details (Name, Email, etc.). This object is passed to both Phase 2 and Phase 3.
3.  **Phase 2 (Email Send):** Consumes user's Name and Email. Produces an email send status (success/failure) and potential error details.
4.  **Phase 3 (CRM Integration):** Consumes all relevant user details from the structured data. Produces a CRM operation status (success/failure), potential CRM contact ID, and error details.
5.  **Phase 4 (Error Handling):** Consumes error messages and relevant input data from any failures in Phase 2 or Phase 3. Produces a log entry and an internal notification.

## Assumptions Made
*   The Google Form is already created and configured to trigger the workflow upon submission (e.g., via webhooks or a polling mechanism).
*   The welcome email template and its content, including any necessary placeholders for dynamic data, are pre-defined and readily accessible.
*   The Gmail account used for sending emails is fully configured, authenticated, and has the necessary permissions.
*   The CRM system's API endpoint, authentication method (e.g., API key, OAuth), and required fields for contact creation are known and accessible.
*   The CRM integration setup and configuration (e.g., API keys, permissions, field definitions within the CRM itself) are completed separately and are out of scope for the *design* of this workflow.
*   The mapping between Google Form fields and CRM fields will be explicitly defined and straightforward, requiring no complex transformations or lookups.
*   The workflow does not need to handle scenarios like duplicate contact checks in the CRM; it is assumed to either always create a new contact or the CRM handles de-duplication internally.
*   "Immediately" for email and "within a reasonable time frame" for CRM implies that these operations are expected to complete within seconds to a few minutes, without significant blocking delays.
